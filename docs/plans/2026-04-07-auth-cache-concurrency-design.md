# 鉴权内存缓存与并发合并设计

**目标**

在不改变现有鉴权语义的前提下，降低热点 token 的重复鉴权开销，并避免同一 token 的并发请求击穿鉴权上游。

**范围**

- 新增 Worker isolate 级鉴权内存缓存
- 新增同 token 的鉴权 in-flight 合并
- 不改动上传缓存策略
- 不改动鉴权错误码和成功语义

**设计**

1. 请求进入后先解析 bearer token，并优先查询 isolate 内存缓存。
2. 若内存缓存命中，直接视为鉴权成功，不再读取 KV，也不再请求鉴权上游。
3. 若内存缓存未命中，再查 KV。
4. 若 KV 命中，回填内存缓存，并视为鉴权成功。
5. 若 KV 未命中，则按 `auth:{sha256(token)}` 在内存中查找 in-flight Promise。
6. 若已有同 token 鉴权进行中，则复用该 Promise。
7. 若没有，则创建新的鉴权 Promise，请求鉴权上游；成功后写入 KV，并回填内存缓存；结束后从 in-flight Map 删除。

**边界**

- 内存缓存仅对当前 isolate 生效，不跨实例共享。
- 内存缓存与当前 KV 一样，正缓存不设置 TTL；token 被上游撤销后，行为与现状一致。
- in-flight 合并同时适用于鉴权成功和鉴权失败场景，但不会缓存失败结果。

**测试**

- 第二次相同 token 请求命中内存缓存，跳过 KV 和鉴权上游
- 两个并发相同 token 请求只触发一次鉴权上游请求
